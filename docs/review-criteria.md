# Review criteria

Maintainers use this checklist when accepting catalog changes. Entries should be **share-ready** (something a reader can adopt from the link), not in-repo drafts or authoring lessons.

## Correctness

- [ ] Links resolve (repository, docs, or relative path in this repo).
- [ ] Type is accurate (tool vs function vs pipe vs integration).
- [ ] No misleading claims; experimental items are labeled.

## Fit for Open WebUI

- [ ] The description states how it connects to Open WebUI (native tool, function, pipe, or external service used from a tool).
- [ ] If it is framework-agnostic, the entry still clarifies the Open WebUI touchpoint.

## Maintenance and safety

- [ ] Prefer items with a visible license or terms when linking code.
- [ ] Security-sensitive tools (browser automation, arbitrary code) include a short risk note.
- [ ] Unmaintained but historically useful entries are allowed with **Status: Unmaintained**.

## Duplicates

- [ ] Merge with an existing entry instead of duplicating, unless the new link adds distinct value (e.g. fork with active maintenance).

## Style

- [ ] English prose for catalog-facing text.
- [ ] Follows [submission-guidelines.md](submission-guidelines.md) format.
